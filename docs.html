Internal Docs

Technical reference for the CraftyForms headless data-capture platform.

Overview

CraftyForms is a headless data-capture platform. It lets you build, publish, and collect responses for any kind of form — without coupling the data model to how it looks or behaves on screen.

Data Flow

  1. Form definitions are stored as JSON in the form_schemas Supabase table.
  2. The Builder (FormBuilder.js) creates and edits definitions.
  3. The Filler (SharedForm.js) serves the public URL, collects answers, and stores them in form_data.
  4. The Dashboard (FormGenerator.js) lists forms, shows analytics, and provides share controls.
  5. The Server handles email, webhooks, incoming/outgoing data APIs, payments, and the MCP integration.

Form Builder

Creating and Editing a Form

Navigate to /builder. The page title updates to Builder | CraftyForms.in. If the URL contains ?editFormId=<uuid> or the router state carries editFormId, the builder loads that form from Supabase and populates all fields.

Ownership is enforced: if the logged-in user is not the form owner, the builder checks form_collaborators for an editor role. Viewers are redirected to the dashboard.

A New Form flow triggered with location.state.newForm = true wipes all builder state (including localStorage keys fb_formTitle, fb_formFields, fb_editingFormId) before rendering so stale data never leaks between sessions.

Options Sources for Choice Fields

Radio, checkbox, and dropdown fields can pull options from four sources:

  • Manual — comma-separated values entered in the builder. Paste-normalisation strips numbered lists, bullets, and excess whitespace automatically.
  • Google Sheet — a spreadsheet ID + range (e.g. Sheet1!A:A). The filler fetches live values at render time; falls back to Google gviz if the primary server is unreachable.
  • Another form — picks unique values from a specific field (by property key) from another form's form_data submissions.
  • Data source — references a registered data_sources row (resolver type: supabase_table, http, or static).

Field-Level Options

  • Required — enforced on submit.
  • Default Hidden — field is hidden until a logic rule shows it.
  • Prefill last answer — if the user previously submitted this form, pre-fills the field with their last answer.
  • Prefill from another form — resolves a field value from a prior submission of a different form.
  • Validation — per-field rules (min/max length, pattern, etc.) stored in a validation object.
  • Allow "Other" — (radio/checkbox) adds a free-text "Other" option.
  • Data-source validation — checks submitted value exists in a named data source.
  • Answer piping — use @{Field Label} syntax in a field's label or description to embed a previous answer at render time.

Logic Rules

Logic rules are stored as an array under schema.v2.logicRules. Each rule has a trigger (condition or condition group) and one or more actions.

Condition operatorAction types
eq neq gt gte lt lte
contains not_contains starts_with ends_with
is_empty is_not_empty		 show hide require unrequire
set_value skip_to submit show_message
trigger_webhook send_email

Condition groups support nested all / any matching. Rules are evaluated in order by the LogicEngine pure function; hidden fields keep their value in FormData but are excluded from validation.

The AI Logic Assistant can suggest rules in plain English. Type a description in the AI panel (logic mode), and the server returns a LogicRule JSON preview. Clicking "Add" appends it.

Settings (v2)

The Settings tab in the builder is organised into accordion sections. All values are persisted to localStorage as the user types and flushed to form_schemas on publish or draft-save.

Renderer

  • Renderer — Classic (default), Card, Chat, or Voice.
  • Card auto-advance — after a choice is made in Card mode, automatically advance to the next card.
  • Show progress bar — display a progress indicator.
  • Auto-submit — submit automatically once all required fields are filled (800 ms debounce).
  • Initial view — whether to show the form or the submissions list when the filler lands on the URL.

Spam Protection

  • CAPTCHA — hcaptcha, recaptcha, or turnstile. Token is verified before submit.
  • Rate limit per IP — max submissions per IP per hour (enforced server-side in the submit function).

Branding

  • Hide Crafty branding — removes the "Powered by Crafty" footer (Pro plan).
  • Brand color — hex value applied to buttons and accents on the filler page via CSS variables.
  • Logo URL — custom logo shown at the top of the filler.
  • Custom CSS / Custom JS — injected into the filler page head.

Access & Submissions

  • Force login — filler must be authenticated. Automatically enabled when Allow multiple submissions is off or Show past entries is on.
  • Allow multiple submissions — if off, each user can submit only once.
  • Show past entries — filler can see their own previous submissions.
  • Allow filler edit — filler can edit their last submission (via ?editId= URL param).
  • Show edit link — display an edit link on the success screen.
  • Allow offline — enable offline submission queue (IndexedDB / localStorage).
  • Close at — ISO datetime; the form auto-closes at this time.
  • Max submissions — auto-close after N responses.
  • Is closed — manual close toggle. Changes propagate immediately to form_schemas and form_versions without requiring a re-publish.
  • Success message — shown on the filler's success screen.
  • Redirect URL — redirect the filler here after a successful submit.

Webhooks

  • Outgoing webhook — on every submit, the filler POSTs to /outgoing-webhook/:formId/:secret. Multiple secrets supported (up to plan limit).
  • Webhook forward URL — the server re-POSTs the payload to this URL (e.g. n8n, Zapier, Make).
  • Incoming API — external systems can push data to /api/incoming-webhook/:apiKey/:formId. Multiple per-form API keys supported.

Google Sheets

Link a Google Sheet via the builder's Sheet picker. The spreadsheet ID is stored in the form_spreadsheets table. On every submission the server appends a row. The spreadsheet modal accepts a full Google Sheets URL or a raw spreadsheet ID (auto-detected).

Publish Flow

  1. Builder validates: title required, at least one field required.
  2. If running in an iframe or popup and user is not logged in, a compact login form appears inline.
  3. Email/password users must have confirmed their email (email_confirmed_at must be set).
  4. Fields are serialised into a JSON-Schema properties map with a stable field_order array. Each key is derived as {label_snake_case}_{index}.
  5. The full schema (including v2 settings) is upserted into form_schemas. A matching row is upserted into form_versions (with is_live = true).
  6. The publish success screen shows the shareable link. Embedded/popup mode posts a CRAFTY_FORM_PUBLISHED message to the parent window and closes.

Auto-save & Undo / Redo

  • Auto-save — debounced 1500 ms after any change. Saves to localStorage; shows a status indicator (saving… / saved / error).
  • Draft save — persists to Supabase without triggering the publish flow. If not logged in, saves locally only.
  • Undo / Redo — Ctrl+Z / Ctrl+Shift+Z. History stack is in-memory (up to 50 steps). Snapshots include fields + rendererKey.
  • Drag to reorder — drag handle on each field card. Updates the formFields array and adjusts editingFieldIndex accordingly.

Version History

Clicking the History button in the builder fetches activity_logs for the current form, ordered by created_at descending. Any entry with an attached schema snapshot can be restored back into the builder (no automatic re-publish).

Collaborators

Form owners can add collaborators (by email) with roles editor or viewer via the Collaborators modal in the builder. Stored in the form_collaborators table. Editors can open the builder; viewers have read-only access.

AI Assist

A sparkle button in the builder top bar opens the AI panel with two modes:

  • Schema generation — describe a form in plain English; the server calls the AI API and returns a full FormDefinition JSON loaded into the builder.
  • Logic assistant — describe a conditional behaviour; the server returns LogicRule JSON objects shown as preview cards. Clicking "Add" appends each rule.

The AI layer is additive — removing it does not affect the builder's core functionality.

Custom Types & Custom Widgets

Custom field types are stored per-user in custom_field_types (type_key, label, icon, base_type, default_config). They wrap a base type with preset configuration (options, min/max, placeholder).

Custom widgets are stored in custom_widgets (widget_key, label, icon, code). The code block must push a component onto window.__CraftyWidgetQueue. If the code is a plain function without the queue push, the builder auto-wraps it. Widget key is auto-derived from the function name (CamelCase → snake_case) or from the label.

Form Filler (SharedForm)

Public URL

Forms are served at /shared-form/:id. The ID suffix ff (e.g. /shared-form/{uuid}ff) is a filler-mode flag — the two trailing characters are stripped before use and the view is forced to filler mode (hides the data-explorer tab).

On load the component resolves the effective schema: it first checks form_versions for a live version (is_live = true); if absent it falls back to the raw form_schemas row. The page title is updated to the form title.

URL Prefill

Any query parameter whose key matches a field's property key is pre-filled on load. Reserved params (editId, mode) are skipped. Existing draft data takes precedence over URL params.

Renderers

KeyBehaviour
classic Default. All fields shown in a single scrollable column. Validates on submit.
card One field per screen. Press Enter to advance. 300 ms animated auto-advance after a choice field is answered (if cardAutoAdvance is on). Progress bar at the top.
chat Conversational bubbles. Bot message = field label. User answer appears as an outgoing bubble. Chat history grows as the filler advances through fields.
voice Wraps the Card renderer with Web Speech API. Reads the current field label aloud via TTS. Accepts spoken answers via STT and pre-fills the widget value. Falls back to Card if window.speechSynthesis is unavailable.

CAPTCHA Integration

When v2.captcha is set, the matching CAPTCHA script is injected into <head> at render time.

  • hCaptcha / Turnstile — auto-rendered widget via class-name div (h-captcha / cf-turnstile). Token is delivered to window.__craftyCaptchaVerified.
  • reCAPTCHA v3 — invisible; token obtained via grecaptcha.execute() at submit time.

The CAPTCHA token is passed to the submit function and verified server-side before insertion.

Submission Flow

  1. Client-side validation of required fields (including logic-engine-determined required set).
  2. CAPTCHA token obtained (if configured).
  3. File / image / audio / video fields uploaded to Supabase Storage; URLs stored in formData.
  4. Payment field processed via Stripe (if present) — submission proceeds only on payment success.
  5. Submission inserted into form_data (primary) and optionally into form_submissions (Step App cross-link).
  6. Outgoing webhook fired to /outgoing-webhook/:formId/:secret.
  7. Email notification sent via POST /send-email.
  8. Google Sheets row appended (if spreadsheet linked).
  9. Success screen shown or redirect executed.

Edit Mode

Adding ?editId=<submissionId> to the URL pre-fills the form from that submission. On submit, the existing row is updated rather than inserted (if allowFillerEdit is enabled).

Offline Mode & Draft Saving

When v2.allowOffline is true, submissions that fail due to network error are queued locally and retried when connectivity is restored.

Draft saving — authenticated users: draft is stored as a form_data row with status = 'draft' and restored on next visit. Unauthenticated users: draft is saved to localStorage keyed by form ID.

Dashboard (FormGenerator)

Forms List

The dashboard fetches forms from form_schemas paginated 10 per page (PAGE_SIZE = 10), ordered newest first. Infinite scroll via a "Load more" trigger appends the next page.

A search bar runs a Supabase ilike query on schema->>title. Search is debounced and replaces the paginated list with filtered results.

A Shared filter toggle switches the list to show forms from form_collaborators where the current user has been added as a collaborator.

Forms pending soft-deletion (30-second undo toast) are tracked in UndoContext.pendingFormIds and hidden from the list until confirmed or cancelled.

Share Modal

Clicking the share icon on any form card opens a three-tab modal:

  • Link tab — displays the public filler URL (/shared-form/{id}ff). One-click copy.
  • Embed tab — generates an iframe or JS snippet. Customisable width (default 100%) and height (default 600px). See Embed section for details.
  • Collaborators tab — add / change role / remove collaborators. Stored in form_collaborators. Roles: editor, viewer.

Analytics Summary

For each loaded form, the dashboard fetches aggregate analytics:

  • Total responses — count of rows in form_data for this form ID.
  • Today's responses — count with created_at >= today.
  • Starts — count of rows in an activity-log or session table where the form was opened.

A summary banner above the list shows totals across all forms (not just the loaded page), computed separately with a single aggregation query.

Embedding a Form

iframe Snippet

<iframe
  id="crafty-form-{form-id}"
  src="https://craftyforms.in/shared-form/{form-id}"
  width="100%"
  height="600"
  frameborder="0"
  allow="microphone *; camera *; geolocation *"
  style="border:none; border-radius:12px; display:block;"
  loading="lazy"
></iframe>
<script>
  window.addEventListener('message', function(e) {
    if (e.data && e.data.craftyHeight) {
      var f = document.getElementById('crafty-form-{form-id}');
      if (f) f.style.height = e.data.craftyHeight + 'px';
    }
  });
</script>

JS Snippet

Dynamically creates the iframe and attaches the same postMessage listener:

<div id="crafty-form-{form-id}"></div>
<script>
  (function(){
    var f = document.createElement('iframe');
    f.src = "https://craftyforms.in/shared-form/{form-id}";
    f.style = "width:100%;height:600px;border:none;border-radius:12px;display:block;";
    f.setAttribute('loading', 'lazy');
    f.setAttribute('allow', 'microphone *; camera *; geolocation *');
    document.getElementById('crafty-form-{form-id}').appendChild(f);
    window.addEventListener('message', function(e) {
      if (e.data && e.data.craftyHeight) f.style.height = e.data.craftyHeight + 'px';
    });
  })();
</script>

Auto-resize postMessage

SharedForm posts { craftyHeight: number } to its parent window whenever the form content height changes. The snippet above listens for this message and updates the iframe height, eliminating scroll bars.

Compact Submit Button in Embedded Mode

When the builder is opened inside an iframe or popup (detected via window.self !== window.top or window.opener), the "Publish" action posts a CRAFTY_FORM_PUBLISHED message containing formUrl to the parent window, then closes the popup if applicable. A compact login form is shown inline if the session is blocked by the browser.

Email Notifications

Email is delivered via SendPulse SMTP API (transactional). Required env vars on the server: SENDPULSE_CLIENT_ID, SENDPULSE_CLIENT_SECRET, SENDPULSE_SENDER_EMAIL, SENDPULSE_SENDER_NAME.

Owner Notification

Triggered on every successful submission when v2.notifications.notifyOwner = true.

  • Recipient: v2.notifications.ownerEmail (or notifyOwnerEmail) → fallback to NOTIFY_OWNER_EMAIL env var.
  • Subject: New response: {form title}
  • Body: a custom message (if set) + an HTML table of submitted field/answer pairs.
If notifyOwner is true but no email address is configured anywhere, the server logs a warning and sends nothing. Add an email in form Settings → Notifications, or set the NOTIFY_OWNER_EMAIL env var as a fallback.

Respondent Confirmation

Triggered when v2.notifications.notifyRespondent = true and the submission contains a non-empty value for the field identified by v2.notifications.notifyRespondentFieldId.

  • That field's submitted value is used as the to address — so the field should be an email-type field.
  • Subject: Thanks for your response — {form title}
  • Body: a custom message (if set) or a default "We received your submission for {form title}" message + field summary table.

Custom Message

If v2.notifications.customMessage is set, it replaces the default body text in both the owner and respondent emails. The message is rendered in a highlighted callout block in the HTML email. Newlines in the message are converted to <br> tags.

Welcome Email

POST /send-welcome-email with { email } sends a branded onboarding email to new sign-ups. Called automatically after a user's first authentication.

REST API

All routes require a Pro or API plan and a Bearer token: 

Authorization: Bearer cf_live_<your-token>

Token format: cf_live_ prefix + 48 hex chars. Obtain or regenerate tokens via the dashboard settings or the token management endpoints below.

List all forms owned by the authenticated user.

Query: limit (1–100, default 50)

Response: { data: Form[], has_more, next_cursor, limit }

Fetch a single form by ID.

Update a form. Accepts title, fields, logic, settings.

Logic is stored in v2.logicRules. Settings are merged into v2.

DELETE/v1/forms/:id

Soft-delete (sets status = 'deleted'). Returns 204.

POST/v1/forms/:id/publish

Publish a form. Sets status = 'published' and published_at.

Response: { id, status: 'published', url }

POST/v1/forms/:id/close

Close a form (status = 'closed'). No more submissions accepted.

GET/v1/forms/:id/link

Response: { url, embed } — the public filler URL and a ready-to-paste iframe snippet.

GET/v1/forms/:id/submissions

List submissions for a form. Query: limit (1–100, default 50).

Each submission has id, form_id, status, filled_at, answers[], meta.

POST/v1/forms/:id/submissions

Programmatically create a submission. Body: key/value pairs matching field IDs.

Response: 201 with the created submission object.

GET/v1/forms/:id/submissions/wait

Long-poll for new submissions. Query: timeout (1–60 s, default 30) and since (ISO datetime).

Polls every 2 s until a new submission arrives or the timeout elapses. Response includes timed_out flag.

GET/v1/submissions/:sid

Fetch a single submission by its submission ID.

DELETE/v1/submissions/:sid

Soft-delete a submission (sets _deleted: true in form_data). Returns 204.

GET/v1/account

Returns { user_id, plan, limits } for the authenticated user.

GET/v1/account/tokens

List API tokens (truncated prefixes only) for the current user.

POST/v1/account/tokens

Create a new API token. Body: { name? }.

Response: { token, name, note, user_id }. Store the token securely — it is shown only once.

Delegates to functions/getSheetData.js. Returns row data from a named sheet.

Delegates to functions/updateSheetRow.js. Updates a specific row in a linked Google Sheet.

Delegates to functions/deleteSheetRow.js. Deletes a specific row from a linked Google Sheet.

POST/logFormChange

Delegates to functions/logFormChange.js. Writes an entry to the activity_logs table.

POST/outgoing-webhook/:formId/:secret

Receives an outgoing webhook payload from SharedForm on every submission.

Validates secret against schema.v2.webhookSecret. Enforces Pro/API plan requirement. Stores last 50 submissions in memory. Optionally forwards to webhookForwardUrl.

Plan limits: free = 0, pro = 5, api = 10 active outgoing webhooks.

GET/outgoing-webhook/:formId/:secret

Poll stored submissions. Query: limit (1–50, default 10).

Response: { ok, count, submissions[] }.

POST/api/incoming-webhook/:apiKey

User-level incoming API. Authenticates via user_api_keys table. Auto-creates a form schema on first call if no matching formTitle exists; reuses on subsequent calls.

Body: arbitrary JSON key/value pairs + optional formTitle (defaults to "Imported API Form") and formId (to target a specific form).

Requires Pro or API plan. API plan users are metered per call and per submission.

POST/api/incoming-webhook/:apiKey/:formId

Per-form incoming API. Validates the API key against schema.v2.incomingApiKey stored in the specific form. Always logs to the named form — never auto-creates.

Requires v2.incomingApiEnabled = true. Enforces plan + per-user limit (pro = 5, api = 10 active incoming APIs).

MCP Server (/mcp)

Implements the Model Context Protocol over HTTP + SSE. Requires Pro or API plan. Auth via Authorization: Bearer <token> or ?api_key=<token>.

EndpointDescription
GET /mcpSSE stream — server-to-client notifications (heartbeat every 25 s)
POST /mcpJSON-RPC 2.0 handler — all tool calls go here

OAuth metadata stubs at /.well-known/oauth-authorization-server and POST /register satisfy mcp-remote's OAuth discovery flow without requiring a real OAuth server.

Connection counts are tracked in mcp_connections table. SSE sessions time out after 3 min without a heartbeat; HTTP sessions after 5 min.

Payments API (/payments)

Powered by Dodo Payments. Live keys use https://live.dodopayments.com; all others use the test endpoint.

GET/payments/status

Return current plan for a user. Query: user_id.

Response: { plan, status, billing_interval, current_period_end, credit_balance_usd }

POST/payments/create-checkout

Create a Dodo checkout session. Body: { user_id, email?, name?, plan, tier?, interval? }.

Plans: pro or api. Tiers for pro: payg (default), 20gb, 50gb.

Response: { checkout_url } — redirect the user here.

GET/payments/verify-checkout

Called from /billing/success after Dodo redirects back. Query: session_id, user_id.

Three resolution paths: in-memory pending store → Dodo API → DB fallback.

Response: { plan, verified, source }

GET/payments/portal

Returns a Dodo customer portal URL. Query: user_id.

POST/payments/cancel-subscription

Cancel at end of current period. Body: { user_id }.

POST/payments/webhook

Receives Dodo webhook events (Standard Webhooks spec). Verified via HMAC using DODO_WEBHOOK_SECRET. Updates plan state in Supabase via RPCs (upsert_user_plan, add_api_credit).

GET/payments/test-db

Debug endpoint — tests that all required Supabase RPCs exist. Returns a results map with pass/fail per RPC.
